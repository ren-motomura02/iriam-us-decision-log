# DSPイラスト制作 コミュニケーション分析

**分析日**: 2026-03-04
**分析対象**: 13チャンネル（スムーズ3、リビジョン2、追加情報収集3、AI参照拒否2、既製モデル1、著作権キャラ1、キャンセル1）
**目的**: 2社目（CDG社）へのオンボーディング資料作成のためのノウハウ抽象化

## 1. 制作フロー全体像

すべてのケースに共通する標準フローを以下に整理する。

```
① Welcome（Bot自動メッセージ）
   ↓
② 初回情報収集（IRIAM staff: yumi）
   - 名前、メール、モデル選択（既製 or カスタム）、X連携URL
   ↓
③ イラストチーム自己紹介（Coordinator: Nathan Luk / Hibikiii / Daniel）
   - 現在のステータス（キュー確認中）
   - レビューチェックポイント3段階の説明
   - 注意事項（アーティスト情報非公開、日本拠点、SNS投稿禁止、著作権、AI禁止）
   - ※ コーディネーターが途中で交代する場合あり（例: Nathan→Hibikiii）
   ↓
④ Pre-production（検証フェーズ）
   - デザインがIRIAMルールに違反しないか確認
   - リファレンス画像の著作権確認
     - 原則URL指定（コピペ自体が著作権侵害リスクのため、引用形式を採用）
     - 1箇所のリファレンスにつき複数枚を推奨
     - 自作 or 著作権上問題ないもののみDriveアップロード可（著作権確認を実施）
   - 不足情報があればストリーマーに追加ヒアリング
   ↓
⑤ 制作開始通知
   - カラーラフ予定日を告知
   - レビューフロー・リビジョンルールの詳細説明
   ↓
⑥ Color Rough レビュー（最大2回のリビジョン）
   - 3営業日以内の返答期限
   ↓
⑦ Line Art レビュー（最大1回のリビジョン）
   - 3営業日以内の返答期限
   ↓
⑧ Final Render レビュー（色調整のみ、最大1回）
   - 3営業日以内の返答期限
   ↓
⑨ ファイル納品
   - Google Driveにアップロード（FIU + HRI）
   - イラストレーターのX（Twitter）アカウント共有
   ↓
⑩ デビュー日程調整（IRIAM staff: yumi）
   - #JumpOnIRIAM投稿日 + デビュー日の確認
   ↓
⑪ デビュー後フォローアップ
   - お祝いメッセージ
   - EASデビューミーティングの予約案内
```

## 2. 各フェーズの定型メッセージ構造

### ⑥⑦⑧ レビューチェックポイント共通のメッセージ構成

すべてのチェックポイントで以下の構造が厳守されている：

1. **挨拶 + ドラフト共有の告知**
2. **確認してほしいポイントの明示**（色、ポーズ、衣装詳細、全体の雰囲気）
3. **変更リクエストの受付案内**（このフェーズで何が変更可能かを明示）
4. **📢 Reminder**: 3営業日以内の返答期限
5. **💻 On Generative AI**: AI利用禁止の注意
6. **画像添付**

### リビジョンルール（フェーズ別）

| フェーズ | リビジョン上限 | 変更可能な範囲 |
|---------|-------------|-------------|
| Color Rough | 2回 | 衣装変更、小さなアクセサリー追加、髪の調整 |
| Line Art | 1回 | アクセサリーのサイズ、衣装の細部 |
| Final Render | 1回 | 色の明度・彩度調整のみ。デザイン自体の変更不可 |

## 3. パターン別分析

### パターンA: スムーズに進行したケース（SleepyTimes / #uxdi-eveninglight）

**特徴**:
- ストリーマーのデザインブリーフが十分で、Pre-productionでの追加確認が最小限
- カラーラフで軽微な修正リクエスト（牙の追加、太もも幅の変更）
- ストリーマーの返信が早く、各フェーズが1〜2日で承認

**コミュニケーション上のポイント**:
- 修正リクエストが曖昧なとき（「太ももをもう少し太く」）、コーディネーターが**カラーラフに直接書き込んでもらうよう依頼**した → 明確な指示を引き出すテクニック
- ストリーマーが「こういう牙を付けて」と手書きスケッチを送った → 即座に対応
- 納品後にFIGファイルの透過問題が発生 → 迅速にファイル再アップロードで解決

**所要期間**: 約7週間（12/16チャンネル開設 → 2/3ファイル納品 → 2/13デビュー）

---

### パターンB: リビジョンが発生したケース（ishi / #ishi-iriam）

**特徴**:
- ストリーマーが非常に情熱的で、デザインへのこだわりが強い
- Pre-productionフェーズでの質問が多い（肌色指定、ポーズ選択、ドレス変更、著作権問題、タトゥーの簡略化）
- カラーラフに対するフィードバックが非常に詳細（長文＋TL;DRあり）

**コミュニケーション上のポイント**:

1. **著作権問題の丁寧な説明**: ハローキティのタトゥーが著作権侵害に該当 → 「we can't use copyrighted characters as reference」と明確に説明し、代替案を受け入れ
2. **プラットフォームルール違反への対応**: 露出の多いドレスについて「cleavageの量が問題」と具体的に説明し、ストリーマー自身がブラウスへの変更を提案 → 「どちらでも対応可能」と柔軟に対応
3. **簡略化の事前合意**: 「accessories may be simplified for the illustrator's workflow」を事前に伝え、合意を取ったことで後のフェーズでの不満を回避
4. **ストリーマーの謝罪に対するケア**: 返信が遅れた際に「no need to apologize」「please do not feel rushed」と伝え、心理的プレッシャーを軽減
5. **大量フィードバックの整理**: ストリーマー自身がTL;DRを書いてくれた（これは例外的。通常はコーディネーターが整理する必要がある）

**所要期間**: 約5ヶ月（8/29チャンネル開設 → 2026/1頃に制作完了見込み。待機時間含む）

### パターンF: 著作権キャラクターデザインのケース（Airi / #airi-iriam）

**特徴**:
- 崩壊スターレイル（Honkai Star Rail）のキャラクター「Cyrene」をベースにしたデザイン希望
- 著作権キャラクターのアクセサリーや特徴的なデザイン要素は再現不可
- 2回のカラーラフレビュー、詳細なビジュアル修正指示

**コミュニケーション上のポイント**:
1. **著作権の線引き**: キャラクターの全体的な「雰囲気」は参考にできるが、特定のアクセサリーやアイコニックなデザイン要素は著作権保護対象として再現不可
2. **詳細な修正指示の伝え方**: 目の形、髪の流れ、ブラッシュの位置等の微細な調整をテキストで伝える際、Hex色コード（例: `#FFB7C5`）を使用して色の認識齟齬を防止
3. **時間制約下での判断**: 納期が迫る中、イラストレーターが自身の判断で微調整を行い、最終版を仕上げた → コーディネーターが「イラストレーターの判断で修正した」と事実を伝達

## 4. 横断的な知見

### 4.1 コミュニケーション原則

| 原則 | 具体例 |
|------|--------|
| **曖昧なリクエストには具体的な指示方法を提示** | 「太ももを太くしたい」→「カラーラフに直接線を引いて示してください」 |
| **拒否する前に一旦預かる** | チェーン追加リクエスト → 「チームに確認します」→ 確認後に不可と回答 |
| **制限を先に伝え、期待値を合わせる** | 各フェーズでの変更可能範囲を制作開始時に詳細説明 |
| **簡略化の可能性を事前合意** | 「アクセサリーは簡略化される可能性があります。よろしいですか？」 |
| **ストリーマーの謝罪にはケアで返す** | 「返信遅れてすみません」→「全然大丈夫です、急がないでください」 |
| **著作権問題は具体的に説明** | 「paid commission ≠ copyright transfer」「copyrighted characters (Hello Kitty, Cyrene) は使用不可」。画像のコピペ自体が侵害リスクのためURL引用を原則とし、自作等の場合のみDriveアップロード可 |
| **定型メッセージの構造を統一** | 毎回同じ構成（変更案内→Reminder→AI禁止→画像）で送付 |
| **色指定はHex色コードで行う** | 曖昧な色表現（「もう少し暖色系」）ではなくHex値（例: `#FFB7C5`）を使い認識齟齬を防止 |
| **リファレンスURL共有の方法自体をガイド** | 「画像を右クリック→Copy Image Address」等、具体的な操作手順を伝える |
| **コーディネーターが代理でリファレンス検索** | ストリーマーが適切なリファレンスを見つけられない場合、代わりに検索して提案 |

### 4.2 よくある対応パターン

| シチュエーション | 対応方法 |
|---------------|---------|
| リファレンス画像の著作権管理 | 基本はURL指定（引用扱い）。1箇所のリファレンスにつき複数枚を推奨。自作 or 著作権上問題ないもののみDriveアップロード可（Discord上で著作権確認を実施） |
| デザインがプラットフォームルール違反 | 具体的にどの要素が問題かを説明し、代替案をストリーマーに提案してもらう |
| 元のスペックにない要素の追加リクエスト | 原則不可。「元のデザインスペックの範囲外のため」と説明 |
| 著作権のあるキャラクターの使用 | 「copyrighted characters は reference として使用不可」と明確に伝え、代替を提案 |
| コミッション作品の著作権 | 「paid commission ≠ copyright transfer。著作権移転の明示的な確認はありますか？」 |
| ストリーマーが長期間未返答 | 3営業日 → 自動続行通知 → フォローアップ → 最終通告 → 資格取消 |
| #JumpOnIRIAM投稿が要件不備 | マニュアル再確認を促す。具体的に何が不足しているかは指摘 |
| FIGファイル等の技術的問題 | 再アップロードや代替バージョンを即座に対応 |
| AI生成リファレンスの提出 | 「AI生成画像はリファレンスとして使用不可」と説明し、Pinterest等のURL提出を案内 |
| 著作権ゲームキャラクターのデザイン希望 | 「雰囲気」は参考可だが、特定のアクセサリーやアイコニック要素は再現不可と説明 |
| Google検索リンクでのリファレンス提出 | 「検索結果URLではなく、画像が掲載されているページのURLを共有してください」と案内 |
| Pinterest URLの無効化 | URLが時間経過で無効になるリスクあり。安定した画像ソースを推奨 |
| 既製モデル+カスタム両方の希望 | 「1人1モデルのポリシー」を理由と共に明確に説明 |
| コーディネーター交代 | 新コーディネーターが自己紹介し、これまでの経緯を引き継いで進行 |
| 時間制約下での微調整 | イラストレーターが自身の判断で微調整を実施 → その旨をストリーマーに伝達 |

| 役割 | 担当者例 | 主な責任 |
|------|---------|---------|
| **IRIAM Staff** | yumi | 初回情報収集、デビュー日程調整、#JumpOnIRIAM確認、デビュー後フォロー |

### 4.4 注意すべきリスクポイント

1. **著作権確認の徹底**: リファレンス画像は原則URL指定（画像のコピー&ペースト自体が著作権侵害の可能性があるため、URLでの引用形式を採用）。1箇所につき複数枚のリファレンスを推奨。自作イラストや著作権上問題ないもののみGoogle Driveへの直接アップロードを許可し、その場合はDiscord上で著作権の確認を行う。コミッション作品の場合は著作権移転の明示的な確認が必要
2. **プラットフォームルール適合**: 露出が多い衣装デザインはPre-productionで早期に指摘し、制作開始前に解決する
3. **デザインスペック外の追加リクエスト**: 制作開始後の大幅な変更はイラストレーターの負荷を増大させる。「元のスペック範囲内」という原則を堅持
4. **未返答ストリーマーへの対応**: 段階的エスカレーションの手順を明確化し、属人的な判断を避ける
5. **AI生成リファレンスの排除**: すべてのリファレンス画像がAI生成でないことを確認。AI生成と判断された場合は受け付けない（13チャンネル中2件で発生）
6. **リファレンスURL形式の問題**: Google検索結果のURL（特定画像を指さない）やPinterest URL（時間経過で無効化）はトラブルの原因になる。画像が掲載されているページの直接URLを指導する
7. **プラットフォーム技術制約**: IRIAMでは目を覆うデザイン（大きな眼鏡等）がサポートされない。デザイン確認時にプラットフォーム技術要件も合わせてチェックする
8. **コーディネーター交代時の引き継ぎ**: 途中でコーディネーターが交代する場合、経緯の把握と新任の自己紹介を確実に行う

### CDG社に特に注意を促すべき点

**コミュニケーション手法**:
1. **曖昧なフィードバックへの対処法**: ストリーマーに「カラーラフに直接書き込んで示す」よう依頼する手法
2. **拒否時のコミュニケーション**: 即座に「No」と言わず、一旦預かって確認後に丁寧に説明する
3. **デザインスペック外リクエストの断り方**: 「原則不可だがチームに確認する」→ 確認後に理由と共に不可を伝える
4. **ストリーマーの感情へのケア**: 遅延や謝罪に対して「大丈夫です」「急がないでください」と伝える重要性
5. **色指定の精度向上**: Hex色コードの活用を推奨。「もう少し暖色系」等の曖昧な指示による手戻りを防止

**Pre-productionでよくある問題（13チャンネルから抽出）**:
6. **AI生成リファレンス**: 13件中2件で発生。即座に拒否し、代替としてPinterest等のURL提出を案内する
7. **Google検索リンク**: 画像検索結果のURLは特定画像を示さない。「画像が掲載されているページのURLを直接共有」と具体的に指導
8. **Pinterest URLの不安定性**: 時間経過でリンク無効化のリスクあり。可能な限り安定した画像ソースを推奨
9. **著作権ゲームキャラクター**: 「雰囲気」の参考は可だが、特定アクセサリーやアイコニック要素の再現は不可。線引きを事前に明確化
10. **URL共有方法のガイド**: ITリテラシーの低いストリーマーに対して、画像URLの取得方法自体を具体的に説明する必要がある場合がある

**制作進行上の留意点**:
11. **コーディネーターの代理対応**: ストリーマーが適切なリファレンスを見つけられない場合、コーディネーターが代わりに検索・提案する柔軟性が重要
12. **イラストレーターからの質問中継**: イラストレーターが制作中に発生した疑問をコーディネーター経由でストリーマーに確認するフロー
13. **時間制約下の対応**: 納期が迫る場合、イラストレーターの判断による微調整を許容し、その旨をストリーマーに伝達
